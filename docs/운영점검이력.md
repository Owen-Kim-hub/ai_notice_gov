# 주간 운영점검 이력

이 문서는 공고 추출 시스템(WMIT)의 **주간 운영점검 및 조치 이력**을 기록한다.
주 1회 점검을 원칙으로 하며, 매 점검 시 아래 두 가지를 확인하고 발견된 문제·원인·조치를
누적 기록한다.
- **점검 A — 공고 추출**: 12개 연동 포털에서 공고가 정상 추출되는지
- **점검 B — 포털 링크**: "연동 및 보정 포털 리스트"의 16개 링크 아이콘이 올바르게 연결되는지

## 점검 A — 공고 추출

1. **빈 키워드(`""`)** + 기간을 주고 전 포털 추출을 돌려 **0건 포털**을 찾는다.
   빈 키워드면 키워드 필터 없이 기간 내 **전체 공고**를 가져오므로, "특정 키워드가 그
   기간에 없어서 0건"인 거짓 양성이 생기지 않는다. 즉 0건 = 진짜 문제 신호.
   - 빠른 점검: `npx tsx` 로 `scrapeAllPortals("YYYY-06-01", "YYYY-06-30", "")` 직접 호출
   - 라이브 점검: 개발 서버(`npm run dev`) 기동 후 `POST /api/extract` (keywords 생략 또는 `[]`)
   - (동작 원리: 빈 키워드는 `appendQuery`가 검색 파라미터를 빼 포털이 전체 목록을 반환하고,
     `matchesKeyword("")`가 모두 통과시켜 → 기간 필터만 적용된다.)
2. 0건 포털은 ① 사이트 접근 가능 여부 ② 파서가 실제 데이터를 뽑는지 개별 진단한다.
3. 원인을 찾으면 `server/portalScraper.ts` 등에서 수정 → 재검증 → 커밋.
4. 점검 결과를 이 문서 상단 요약표와 상세 섹션에 기록한다.

> 참고: 제외 상태 4개 포털(IRIS 접수예정, HTDream, KHIDI ×2)은 추출 대상이 아니다(점검 A 제외). 단, 링크 점검(점검 B)에는 16개 전부 포함된다.
> 6개월 등 넓은 범위는 소프트 타임버짓(45초)에 걸려 하위 우선순위 포털이 건너뛰어질 수 있다(설계상 정상). 추세 점검은 30일 범위 권장.

## 점검 B — 포털 링크 아이콘

`src/components/PortalList.tsx`("연동 및 보정 포털 리스트")는 16개 포털 각각을
`<a href={url} target="_blank">`로 **사용자 브라우저에서 직접** 연다(`/api/open` 화이트리스트를
거치지 않음). 따라서 점검은 16개 URL의 도달성을 확인하는 것이다.

1. `node scripts/check-portal-links.mjs` 실행 → 16개 URL을 GET으로 점검(HTTP 2xx + 본문 크기).
   URL은 `src/constants.ts`의 `PORTALS_LIST`에서 자동으로 읽으므로 단일 출처가 유지된다.
2. "확인필요"로 표시된 링크는 **실제 브라우저에서 한번 더 확인**한다. 링크 아이콘은
   브라우저(OS DNS 리졸버)로 열리는데, 이 스크립트는 Node의 DNS 리졸버를 쓰므로 일부
   포털(특히 CNAME 구조)에서 `EAI_AGAIN` 같은 Node 한정 거짓 실패가 날 수 있다.
   - 예: htdream.kr은 `www→web3.htdream.kr` CNAME 구조라 Node fetch가 EAI_AGAIN으로 실패하지만
     `nslookup`(OS)은 정상 해석한다(2026-06-30 점검 사례). 단, htdream은 원래 불안정해 추출제외 상태.
3. URL이 실제로 바뀐(사이트 개편 등) 경우 `src/constants.ts`의 해당 url을 갱신한다.

**발견된 문제**

1. **과학기술정보통신부 (msit)** — 추출 0건
   - 원인 ① 검색 시 `searchTxt` 파라미터를 함께 보내면 서버가 빈(템플릿) 페이지 반환
   - 원인 ② 사이트가 JS 동적 렌더링 구조(`unescape('제목')` + `td_REG_DT` 날짜 + `fn_detail(번호)`)로 변경 → 기존 `fn_detail+unescape` 단일 패턴이 함수 정의부에만 매칭돼 1건만 잡힘
   - 원인 ③ 날짜 형식 `2026. 6. 30`(구분자 공백)을 `normalizeDate`가 정규화 실패
   - 조치: searchTxt 제거 / JS 렌더 구조 파싱으로 재작성 / `normalizeDate` 공백 허용

2. **범부처전주기의료기기 (kmdf)** — 추출 0건
   - 원인: generic 파서가 제목을 링크 *앞쪽*에서 찾는데, 실제 제목은 `<a href="...">제목</a>`처럼 링크 *뒤*에 있어 빈 값
   - 조치: 링크 뒤 텍스트 우선 추출 + 날짜 탐색 범위 확대

**기타 조치**

**포털 링크 점검 (점검 B)** — `node scripts/check-portal-links.mjs`, 결과 15/16
- 16개 중 15개 HTTP 200 정상.
- **보건의료기술 종합정보시스템 (htdream, 추출제외)**: Node fetch가 `EAI_AGAIN`(DNS) 실패.
  단 `nslookup`은 정상 해석(210.179.230.213) → `www→web3.htdream.kr` CNAME 구조에 대한
  Node 리졸버 한정 문제로 판단. 브라우저 링크 아이콘은 OS 리졸버를 쓰므로 실제로는 열릴 가능성 높음.
